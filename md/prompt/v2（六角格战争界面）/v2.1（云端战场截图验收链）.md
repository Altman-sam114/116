# v2.1 云端战场截图验收链

## 版本号与分配依据

v2.1。v2.0 已在 `main` 完成六角格战场表现层拆分，并由 run `29220790248`、attempt `1`、artifact `ww2tactics-ci-v2.0-main-277e6e0-run29220790248-attempt1` 证明 static、rules smoke 和 build-for-testing 成功；但该结果包没有运行截图，无法证明首屏视觉质量。v2.1 在同一阶段补齐云端视觉证据链。

## 背景与目标

人工明确要求不做本地 build、模拟器或视觉测试，所有重验证都在云端执行。当前 CI 只构建 app 和测试 bundle。此轮让 GitHub Actions 在 build 成功后启动可用 iOS Simulator，安装并启动 `WW2Tactics`，生成首屏战场截图，并把截图状态纳入 manifest、failure summary、JUnit 和最终门禁。

## 非目标

- 不修改 SwiftUI 画面、`GameState`、`GameModels`、地图坐标或规则。
- 不新增 UI test target、第三方 snapshot 库、图片基线或像素差阈值。
- 不在本机运行 Xcode、Simulator、swiftc、smoke、XCTest 或截图命令。
- 不触碰工作树已有的 `project.pbxproj` 与 `AGENT.md` 改动。

## 架构依据与实现步骤

1. 保留现有 static、rules smoke 和 generic iOS Simulator build-for-testing。
2. build 成功后从 `simctl list devices available --json` 选择最新 runtime，优先 iPad Pro、其次其他 iPad、最后 iPhone。
3. 复用 build-for-testing 生成的 `WW2Tactics.app`，启动模拟器、安装 app、设置确定性状态栏、启动 bundle `com.easytech.prototype.WW2Tactics`，等待首屏稳定后截图。
4. 生成 `ci-results/battlefield-screenshot.png` 和 `ci-results/battlefield-screenshot.log`；任何 boot/install/launch/capture/空文件失败均令 screenshot outcome 失败。
5. manifest 记录 screenshot outcome、路径、设备与 runtime；failure summary 和 JUnit 加入 screenshot 检查；最终门禁同时要求 static、smoke、build、screenshot 成功。
6. artifact 继续未加密上传完整 `ci-results`。下载后必须查看真实 PNG，确认不是 SpringBoard、黑屏、崩溃画面或空白图。

## 测试与云端验收

本地不运行任何构建或测试。只允许 Git diff 范围检查和 YAML 静态解析，且不得称为测试。提交信息为 `v2.1: 增加云端战场截图验收`，直接 push `origin/main`。

最新 run 必须满足：manifest 的 branch/SHA/run/attempt 与 `origin/main` 一致；static、smoke、build、screenshot 全部 success；JUnit 为 4 项 0 failures；`.xcresult`、截图、截图日志均存在；截图像素尺寸大于零且人工查看为已启动的 WW2Tactics 战场。若失败，只按最新 artifact 日志追加修复，不使用旧 run 冒充。

## 文档更新要求

更新测试分层、CI 数据流、流程图、prompt 索引和 update log；同时补记 v2.0 最终成功 run，清除“等待验收”的过期表述。

## 风险与禁止项

- runner 设备名和 runtime 会变化，必须从 JSON 动态选择，不能写死 UDID。
- 截图文件存在不等于 app 正常，验收必须实际查看图片。
- 模拟器必须在步骤结束时 shutdown；runner 临时环境不得成为新的持久依赖。
- 禁止把本机输出、旧截图或官方参考图冒充本轮截图。
- 禁止回滚、暂存或提交用户已有工作树改动。
