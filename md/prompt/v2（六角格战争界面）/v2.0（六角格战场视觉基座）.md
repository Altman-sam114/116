## 背景

当前项目已有完整可玩的六角格规则链，包括地图点按/鼠标主副键、MOVE/ATK/POS、路线、威胁、据点、反制、态势与 AI 回放。问题主要在表现层：`ContentView.swift` 承担外壳、地图、HUD、单位、Inspector 和所有结果面板，文件边界失控；地图格之间的描边和阴影强化了“拼块感”，没有形成连续战区；单位虽有兵种轮廓，但尺寸小、信息挤在一起，仍像调试棋子；顶栏和地图内 HUD 占用较多视觉空间，选中单位后的核心信息分散。

人工提供的《Glory of Generals 3》官方截图 `/private/tmp/gog3-ipad-1.jpg` 至 `-4.jpg` 只作为构图与信息层级参考。可借鉴的通用特征是：地图占据绝大多数首屏、地貌跨格连续、单位主体直接落在地图上、阵营底座和血条近身显示、将领头像附着于单位、顶栏很薄、命令沿屏幕边缘布置、选中后只展开当前对象信息。不得复制截图、角色肖像、商标、图标、纹理或其他版权资产；全部视觉必须由原创 SwiftUI `Shape`、`Canvas`、颜色、渐变和 SF Symbols 构成。

## 本轮目标

1. 把地图、HUD、单位视觉从 8,839 行 `ContentView.swift` 抽离为职责明确的独立 Swift 文件，并加入 app target；`ContentView.swift` 只保留根布局、Inspector 及未纳入本轮的侧栏/战报组件。
2. 让六角格战场成为横竖屏首屏的绝对主视觉：连续地貌优先，六角边界退居交互辅助层，地图不再读成一组独立卡片。
3. 用原创 SwiftUI 绘制可一眼辨认的坦克、步兵、炮兵和侦察单位轮廓；保留阵营底座、HP、军衔、行动/补给/战术状态，并为有将领单位提供稳定的头像徽章。
4. 将首屏 chrome 收敛为薄顶栏、地图边缘快捷命令和选中后的紧凑详情，降低遮挡且不移除任何现有可执行入口。
5. 完整保持 `GameState` 行为、地图 marker 数据源、六角坐标布局、命中区域和点按/主键/副键调用链不变。
6. 横屏 iPad、竖屏 iPhone 和窄宽断点都可用；主要操作至少 44x44pt，支持 Dynamic Type、VoiceOver 和 Reduce Motion。
7. 只使用现有系统框架和原创代码，不引入第三方依赖、下载资产或复杂资产管线。

- 不新增战斗动画系统、粒子、音效、真实 3D 模型、图片贴图或将领真人肖像。
- 不重新设计战役选择页、科技页、主菜单或营销页。
- 不改变地图尺寸、`HexCoordinate` 轴向坐标、`position(for:)` 公式、`Hexagon.path(in:)`、tile frame、`HexInputReader.hitShape`、滚动焦点或地图缩放语义。
- 不用 View 本地状态复制 `GameState` 的选中、焦点、播放、命令或规则状态。
- 不删除 MOVE/ATK/POS/NEXT/OBJ/THR/INT/PRS/RSP/AI 等既有反馈，也不以更美观为由隐藏不可用目标提示。

## 当前架构依据

- `GameModels.swift` 的 `HexCoordinate` 使用轴向 `q/r` 坐标并提供六邻接与 hex distance；`TerrainKind` 提供 plains/forest/city/mountain/snow/river/road；`TerrainTile` 只保存坐标、地形、据点名和归属；`BattleUnit` 保存兵种、阵营、位置、HP、将领、经验、士气和行动/战术状态。本轮只能消费这些既有数据。
- `GameState.swift` 是所有规则与命令的唯一状态机。地图 View 只能读取派生状态，并把用户意图转发给 `handleTap(on:)`、`handlePrimaryAction(on:)`、`handleSecondaryAction(on:)` 等现有入口。
- 当前 `HexMapView` 使用 `tileWidth=86`、`tileHeight=74`，位置公式为 `q * width * 0.78 + r * width * 0.39` 与 `r * height * 0.76`；每格 `HexInputReader` 的命中路径来自同尺寸 `Hexagon`。这些是本轮行为兼容基线。
- 当前 `HexTileView` 已有 marker 顶/底槽位、角落命令标记、完整 accessibility label 和状态边框优先级。视觉重构必须复用这些数据与优先级，不得重新发明 marker 状态机。
- `ContentView.swift` 当前主要边界：1-813 为主题/根布局/顶栏/地图 chrome/HUD；1,894-3,423 为地图、tile、marker 和单位；8,411 行后为 `Hexagon` 与 `HexInputReader`。本轮按组件依赖迁移，不做 8,839 行全文件重写。

### 3. 重建连续战区地貌

1. `MapGridBackdrop` 改为完整战区底图基底：按当前场景面积铺设低对比纸质/战场底色、轻微方向性光照和稀疏原创纹理。可用 `Canvas` 的确定性绘制、SwiftUI `Shape` 和渐变；不得使用随机数造成每次 body 重算闪动。
2. tile 仍保持六角 content shape 与命中，但基础填充取消厚重独立阴影和卡片式亮边。相邻 tile 应视觉相接，边界只保留低对比细线；选中、MOVE、ATK、POS 等交互状态再提高边线对比。
3. 为七类地形设计可跨格阅读的原创语汇：平原为低对比田野纹理，森林为树簇/深色覆盖，城市为道路与建筑块，山地为脊线，雪地为冷色覆层，河流为连续水道感，公路为贯穿路径感。不得给每格放一个巨大 SF Symbol 作为主要地貌。
4. 连续感优先于严格模拟真实河网。使用相邻 `TerrainTile` 的同类邻接信息做纯展示连接时，只能在 View 内从 `game.tiles` 派生，不得写入状态或更改移动成本；邻接方向必须来自现有 `HexCoordinate.neighbors`。
5. 据点用小型旗帜/建筑群/名称牌体现，归属色只做局部环、旗帜或微弱区域染色，避免整格高饱和阵营色破坏地貌。
6. marker 顶/底栈与角落命令槽位继续工作；地图材质不能压低命令对比，也不能遮挡单位和据点。

### 4. 重建单位模型与将领徽章

1. 在 `BattlefieldUnitViews.swift` 为 `UnitKind.infantry/tank/artillery/recon` 提供四种独立 `Shape` 或组合轮廓：坦克必须可见车体、履带/炮塔/炮管；步兵为多人或持枪剪影；炮兵为炮架、炮轮与炮管；侦察为轮式装甲车/侦察车。不得仅靠 `INF/TNK/ART/REC` 文本区分。
2. 轮廓使用中性军械色，阵营用清晰底座环/楔形底盘区分；盟军与轴心除颜色外再用不同底座符号或边缘样式，保证色觉障碍下仍可识别。
3. HP 条贴近单位底座，宽度稳定且不随数值改变布局；危险 HP 使用高对比色。军衔、CUT、战术状态和已行动状态按既有字段保留，但降为角标，不与单位主体争夺中心。
4. 有 `Commander` 时显示原创“将领头像徽章”：使用人像 SF Symbol 或由 Shape 组成的通用军官剪影、阵营环和星级/姓名首字，不得伪造历史人物肖像。无将领时不保留空洞占位。
5. 地图上的单位主体在常规缩放下必须一眼可辨；编队条、图例、攻击目标和单位详情复用同一 visual language，但允许 compact 参数降低细节，不能复制四套不同绘制实现。
6. `UnitCounter` 的 accessibility label 必须包含单位名、兵种、阵营、HP、将领、军衔、补给/战术/已行动状态；视觉子元素合并为一个可访问元素，避免重复朗读。

### 5. 收敛顶栏、边缘命令和选中详情

1. 顶栏改为单行优先的薄状态带：战役名/回合/阵营/结束回合为一级信息，其余战力、星级、据点、指令点在空间不足时横向滚动或紧凑折叠，不允许挤压地图高度。结束回合保持至少 44x44pt。
2. 地图必须是主区域。横屏 regular/large width 下地图占可用宽度的明显多数，Inspector 保持现有约 308-336pt 边栏但不得扩宽；竖屏下先保证地图有稳定最小高度，再显示现有 Inspector。
3. MOVE/ATK/POS/OBJ/THR 等现有快捷命令改为贴地图边缘的紧凑命令带或图标按钮，使用 SF Symbols、短标签和 tooltip/accessibility hint；每个命令的实际 action、disabled 和数据源不变，点击区至少 44pt。
4. 未选中单位时不显示空的大型动作面板；选中后显示紧凑详情，包括单位轮廓、名称、HP/行动状态和当前命令预览。不要另造规则摘要，不要复制 Inspector 的长面板。
5. 地图 chrome 使用安全区和固定边缘约束，不能盖住顶栏、ObjectiveJumpDock、单位主体或 marker。所有固定格式控件需有稳定 frame/minWidth/aspectRatio，避免 Dynamic Type 导致地图跳动。

### 6. 响应式、可访问性和动效降级

1. 保持现有 `>=900`、`>=760` 与窄屏分支的意图，可调整数值但必须同时覆盖横屏 iPad、竖屏 iPhone、窄宽窗口；任何横向状态带都应可滚动或使用 `ViewThatFits`，不得文字重叠。
2. 主要按钮、编队单位、地图快捷命令、结束回合、重开和选中详情操作至少 44x44pt；纯显示徽章不必伪装为按钮。
3. 使用语义字体和合理换行；不得用 viewport 宽度缩放字号。重要名称允许两行，最长战役名/单位名不得越界。
4. VoiceOver 保留地图格完整语义，并为图标命令提供 label/hint/value；装饰性地貌、纹理和单位内部 Shape `accessibilityHidden(true)`，避免噪音。
5. 读取 `@Environment(\.accessibilityReduceMotion)`。滚动聚焦和状态强调在 Reduce Motion 下禁用或改为无动画；不得改变聚焦目标和命令执行。现有 AI 播放状态机不在本轮修改范围。

### 7. 保持行为兼容

逐项核对以下代码事实未变化：

- `HexMapView.position(for:)` 的 tile 几何和 `tileScale` 缩放链。
- 每格 `.frame(width: tileWidth, height: tileHeight)`、`.position`、`.id(tile.coordinate.id)`。
- `HexInputReader` 的 direct touch、primary 和 secondary recognizer，以及三个 action 分别调用 `handleTap`、`handlePrimaryAction`、`handleSecondaryAction`。
- `Hexagon.path(in:)` 与 `hitShape` 使用同一尺寸。
- `ScrollViewReader` 按 `focusedCoordinate.id` 居中；Reduce Motion 只影响是否动画。
- 所有 marker 集合、排序、折叠、border priority、VoiceOver 文案和命令 action/disabled 条件。
- `GameModels.swift`、`GameState.swift` 和规则测试不因纯 UI 重构发生业务变更。

## 验收标准

### 结构

- `ContentView.swift` 不再包含地图、HUD 和单位绘制的主要实现；至少新增 theme、chrome/HUD、map/input、unit visuals 四个职责文件。
- 四个新增文件均在 `WW2Tactics` app target Sources 中，CI build log 实际编译它们。
- `ContentView.swift` 行数显著下降，且没有通过复制留下同名/同职责双份组件。
- `project.pbxproj` 的 v2.0 commit 只增加 target membership，不含人工 `DEVELOPMENT_TEAM` 和 group 顺序差异。

### 视觉与可玩性

- 横屏和竖屏首屏都以六角格战场为主，而非面板或卡片墙；横屏地图占可用宽度明显多数。
- 相邻六角格读成连续战区地貌，七类地形可辨；默认边界不呈现厚阴影卡片墙，选中/命令状态仍清晰。
- 坦克、步兵、炮兵、侦察不读文字也能从轮廓区分；阵营除颜色外还有底座形态差异。
- HP、行动状态、军衔、补给/战术角标和将领徽章清楚但不遮挡单位主体；将领徽章为原创通用头像，不复制历史人物。
- 顶栏薄且信息不溢出；地图边缘快捷命令和选中详情可操作、不挡主要战况。
- MOVE/ATK/POS/OBJ/THR/INT/PRS/RSP/AI 等既有标记仍可读，至少 44pt 的主要点击区、Dynamic Type、VoiceOver 和 Reduce Motion 要求均落实在代码。

### 行为与验证

- `GameModels.swift`、`GameState.swift` 规则和六角命中语义没有变化。
- 点按、鼠标主键、鼠标副键仍分别进入原有 `GameState` 调用链；坐标布局、滚动聚焦与 marker 数据源不变。
- Agent B 没有运行或宣称任何本地 build/typecheck/smoke/模拟器测试，只报告允许的静态范围检查。
- v2.0 已在 `main` 精确提交并 push 到 `origin/main`。
- 最新 GitHub Actions run 的 static checks、rules smoke 和 build-for-testing 成功，生成未加密 artifact。
- Agent C 下载并逐项核对 manifest、failure summary、JUnit、logs、项目结果文件和 `.xcresult`，且 manifest SHA 等于最新 `origin/main`。

## 风险与禁止项

- 风险：跨文件后 `private` 声明不可见。应按最小范围改为默认 internal，不能把表现层 API 无差别改成 `public`。
- 风险：在 tile 上增加复杂 Canvas 可能造成大地图重绘压力。纹理必须确定性、低层数、无每帧随机；优先复用轻量 Shape 和缓存式纯值配置。
- 风险：为了连续地貌改变 tile frame/position 会破坏命中。视觉连接只能在既有几何之上绘制，不能改坐标与 hit shape。
- 风险：单位主体变大会与 marker 冲突。应稳定划分主体、HP/底座和顶/底/角 marker 槽位，不能隐藏 marker。
- 禁止修改规则、AI、移动成本、伤害、据点归属或新增不可测试玩法状态。
- 禁止把核心规则或 marker 生成搬进 View。
- 禁止重写其他侧栏、删除现有入口或把首屏做成静态展示/营销页。
- 禁止第三方库、网络依赖、外部下载、截图切片、版权纹理、真实将领肖像和复杂资产管线。
- 禁止为“像 EasyTech”直接复制官方截图布局像素、品牌、图标或资产；只能学习地图主导、信息密度和层级原则。
- 禁止修改 `HexCoordinate`、`position(for:)`、`Hexagon.path` 或 `HexInputReader` 的行为。
- 禁止回滚、覆盖、暂存或提交人工 `project.pbxproj` 改动。
- 禁止运行本地 build、typecheck、smoke、XCTest 或模拟器测试；禁止把静态 diff 检查称为测试。
- 禁止使用旧 run、旧 artifact 或本地输出冒充 v2.0 云端验收。
- 禁止在 Agent C 未下载并核对最新 artifact 前宣称 v2.0 验收通过。
