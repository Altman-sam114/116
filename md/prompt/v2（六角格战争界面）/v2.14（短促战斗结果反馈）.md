# v2.14 短促战斗结果反馈

## 版本号与分配依据

- 版本号：`v2.14`
- 分配依据：v2.13 已由 `origin/main` commit `3bb90532e075e559b3254671d13f86d7a7714d50`、GitHub Actions run `30073890297`、attempt `1` 和 artifact `ww2tactics-ci-v2.13-main-3bb9053-run30073890297-attempt1` 完成云端验收。攻击聚焦已经清理视觉噪音，但当前开火后的真实伤害仍主要落在 Inspector 结果卡，地图缺少 EasyTech 风格的即时战损回读，因此递增 v2 小版本补齐战斗闭环。

## 背景

v2.13 的 2064x2752 云端截图证明攻击前的火力解算、目标准星、交战轴线和降噪有效。下一步需要让玩家按下“开火”后不必转移视线到侧栏：交战轴线上应短促呈现攻击伤害、反击、双方 HP 前后和交火/击毁结论。现有 `latestCombatResult` 已保存真实结算数据，但没有交战坐标；若攻击后自动切换选中单位或目标被击毁，View 无法可靠恢复原交战位置。

## 目标

1. 为普通攻击结果保存攻击方和防守方执行时的六角坐标，坐标属于结算事实而非 UI 临时状态。
2. `HexMapView` 从 `latestCombatResult` 只读绘制地图级战斗结果叠层。
3. 叠层清晰显示 `HIT -伤害`、防守方 HP 前后；有反击时显示 `RET -伤害`、攻击方 HP 前后。
4. 交战中央显示“交火”或“击毁”结果，并用短轴线/冲击视觉连接双方位置。
5. CI 新增真实执行攻击后的固定截图场景，云端 artifact 能证明 HP 已改变且地图结果反馈可见。

## 非目标

- 不修改伤害、反击、夹击、经验、士气、击毁、追击、选择轮转或 AI 规则。
- 不在 View 重算伤害、反击或生存判定，不复制 `GameState` 公式。
- 不新增计时器、持续动画、自动清除规则或 UI 私有战果副本。
- 不重构 Inspector 的既有 `CombatResultSummaryView`，不做无关页面重排。
- 不改变 tile frame、坐标转换、滚动、缩放、命中或手势。
- 不修改 Xcode project，不引入第三方库、网络依赖或位图资产。
- 不修改或提交用户现有的 `AGENT.md` 删除和 `project.pbxproj` 修改。

## 当前架构依据

- `GameState.attack(...)` 在结算前已持有攻击方与防守方 `BattleUnit`，可把两者执行时 `position` 写入 `CombatResultSummary`。
- `latestCombatResult` 是普通攻击后的结构化单一事实来源，已经与其他结果互斥清理并被 Inspector 使用。
- `HexMapView.position(for:)` 是六角坐标到地图点位的唯一表现层转换，可直接定位战果叠层。
- 攻击结束会调用 `selectNextReadyUnit(...)`，因此不能从 `selectedUnit` / `focusedCoordinate` 推断上一场交战。
- SwiftUI 叠层必须是独立 View、禁止 hit testing、提供组合后的 VoiceOver 战果摘要，并尊重 Reduce Motion；本轮固定态不需要新增动画。

## 实现步骤

1. 在 `CombatResultSummary` 增加 `attackerCoordinate` 与 `defenderCoordinate`，由 `GameState.attack(...)` 使用结算前双方位置填充。
2. 更新既有 XCTest 与 RulesSmokeTest 断言，确认坐标与真实执行目标一致；不改变其他战斗结果字段语义。
3. 在 `BattlefieldMap.swift` 增加独立 `CombatResolutionOverlay`，输入仅为 summary 及已经换算的两端点。
4. 叠层在防守端显示 `HIT -N` 和 `startingHP -> endingHP`，仅在实际反击时于攻击端显示 `RET -N` 和对应 HP；中央显示 `交火`、`击毁` 或 `压制`，击毁态使用爆裂/准星符号而不是仅依赖颜色。
5. 保持文字尺寸稳定、描边与底色有对比，不遮挡双方军械主体和 HP 条；叠层整体 `.allowsHitTesting(false)`，用组合 accessibility label 描述双方战损与结果。
6. `WW2TacticsApp` 增加 `--ci-selected-combat-result`，依次调用 `handleTap(q7,r6)`、`handleSecondaryAction(q9,r6)`、`executeFocusedCommand()`；保留 v2.12/v2.10 旧参数。
7. workflow 把截图场景改为 `selected-combat-result` 并传入新参数，保留 500 KB 门禁、设备/运行时/场景元数据和未加密 artifact。
8. 同步 README、flow、flowchart、test、prompt README 与 update log，记录 v2.13 精确验收证据和 v2.14 等待状态。

## 验收标准

1. 云端截图是攻击后状态，而非 v2.13 的攻击前预览；M10 和目标 HP 均与结算结果一致地下降。
2. 防守端附近清晰显示 `HIT -21` 与 `64 -> 43`；若本场存在反击，攻击端附近显示 `RET -11` 与 `64 -> 53`，具体数值以云端真实规则结果为准。
3. 交战中央能直接读出“交火”或真实击毁结果，反馈不依赖 Inspector 结果卡才能理解。
4. 叠层不遮挡双方单位主体、HP 条、关键 HUD，不改变地图尺寸或输入。
5. 代码审查确认所有数值和结论只读自 `latestCombatResult`，交战位置来自其中的结构化坐标，View 无伤害公式或临时战果状态。
6. 测试覆盖结果坐标；旧攻击预览和接敌预览启动参数仍保留。
7. 最新 SHA/run/attempt 与 artifact manifest 完全一致，所有云端门禁成功，PNG 至少 500 KB 且实际画面非黑屏。

## 风险和禁止项

- 不能用当前 selected/focused 坐标冒充上一场交战坐标，否则攻击后选中轮转会错位。
- 不能只在 Inspector 显示结果或用 `message` 字符串解析战损。
- 不能让大面积卡片盖住地图；反馈应贴近交战位置并保持短促、战术化。
- 不能让颜色成为唯一结果区分，必须同时使用文本或图标。
- 不能以常驻动画制造 CI 截图不确定性。
- 不得触碰用户改动、分支或 PR 流程。
