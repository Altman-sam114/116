# v0.4（规范 Agent C 验收提交）

## 版本号

v0.4

## 版本分配依据

人工要求修改当前 `AGENTS.md` 工作流：Agent C 最终审查通过后自动按版本号 git 提交；不通过则指出问题并退回 Agent B。该改动属于 v0 项目管理阶段的小版本规范更新。

## 背景

当前项目已建立 `AGENTS.md + update_log.md + md/prompt + md/test + md/flow` 的多 Agent 协作体系，但 Agent C 的验收后动作只写到“输出结论”，尚未明确通过后自动提交版本。

## 目标

- 更新 `AGENTS.md` 的 Agent C 工作流。
- 增加“版本提交规则”。
- 更新 `md/flow/flowchart.md` 的 Agent 迭代流程图。
- 更新 `update_log.md` 记录 v0.4。
- 创建本版本 prompt 文件。

## 非目标

- 不修改 Swift 源码。
- 不实际执行 git commit。
- 不改变 Agent A 和 Agent B 的核心职责。

## 当前架构依据

- `AGENTS.md` 是入口规则和多 Agent 工作流。
- `update_log.md` 记录版本和历史决策。
- `md/flow/flowchart.md` 展示 Agent 迭代流程。
- `md/prompt/` 保存版本化提示词。

## 实现步骤

1. 在 `AGENTS.md` 的 Agent C 流程中加入通过/不通过分支。
2. 写明不通过时退回 Agent B，不得提交。
3. 写明通过时按版本号自动创建 git commit。
4. 规范提交信息格式：`vN.x: 简要说明本版本做了什么`。
5. 更新交付格式，要求报告提交哈希和提交说明。
6. 更新禁止项，禁止不通过时提交和模糊提交信息。
7. 更新 `md/flow/flowchart.md` 的 Agent 迭代 Mermaid 图。
8. 更新 `update_log.md`。
9. 运行 `git diff --check`。

## 关键文件

- `AGENTS.md`
- `md/flow/flowchart.md`
- `update_log.md`
- `md/prompt/v0（项目管理）/v0.4（规范AgentC验收提交）.md`

## 测试要求

文档-only 修改至少运行：

```sh
git diff --check
```

## 文档更新要求

- `update_log.md` 追加 v0.4。
- `md/flow/flowchart.md` 的 Agent 迭代流必须显示验收通过后提交、不通过退回 B。

## 验收标准

- Agent C 工作流明确不通过退回 B。
- Agent C 工作流明确通过后自动 git commit。
- 提交信息格式按版本号管理。
- 最终汇报要求包含提交哈希和提交说明。
- `git diff --check` 通过。

## 风险和禁止项

- 不要让 Agent C 在测试缺失或文档不同步时提交。
- 不要使用 `update`、`fix`、`1` 等无意义提交说明。
- 不要实际提交本轮文档变更，除非用户明确要求立即按新规则提交。
