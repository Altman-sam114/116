# v1.16 反制建议排序对比解释

## 版本号

v1.16

## 版本分配依据

当前 `md/prompt/v1（地图操作体验）/` 最新功能版本为 v1.15（反制建议收益解释）。本轮继续沿着 v1 地图操作体验阶段推进，不开启新大版本，因此递增为 v1.16。

## 背景

v1.10 到 v1.15 已经完成敌方意图、反制建议、点选聚焦、地图标记、执行入口桥接和收益解释。玩家现在能看到“该建议有什么收益”，但仍缺少“为什么这条建议排在前面、它相对下一条优势在哪里”的紧凑解释。EasyTech 风格大地图战棋需要让玩家快速理解参谋建议的优先级，而不是只看到一个抽象分数。

## 目标

1. 为 `EnemyThreatCountermeasurePreview` 增加只读排序/对比解释字段，说明建议优先级来自可执行性、击毁、score、路线消耗、执行单位、目标和坐标等既有排序维度。
2. 为反制建议列表生成“相对下一条建议”的紧凑对比提示，至少覆盖第一条与第二条之间的主要排序差异。
3. 在 SwiftUI 反制建议面板中展示排序原因和相邻对比，不改变建议的排序算法、不改变执行逻辑。
4. 补充 XCTest 和规则 smoke test，证明排序解释与现有排序维度一致，并且查询仍然只读无副作用。
5. 同步 README、flow、flowchart、test 和 update_log。

## 非目标

- 不改变 `enemyThreatCountermeasureSort` 的排序规则或 score 公式。
- 不改变敌方意图生成、AI 行动、战斗数值、移动规则、整补成本、胜负条件。
- 不让反制建议行直接执行攻击、移动或整补。
- 不新增第三方库、网络依赖、图片资产或复杂资产管线。
- 不重构项目架构，不迁移 `ObservableObject/@Published`。

## 当前架构依据

- `GameModels.swift` 定义 `EnemyThreatCountermeasurePreview`、收益指标和执行入口模型；本轮新增的排序解释模型也应放在这里，保持纯数据、纯派生，不 import SwiftUI。
- `GameState.swift` 负责生成和排序反制建议；若需要跨建议对比，应由 `GameState` 基于已排序结果生成只读摘要，不能交给 `ContentView` 复刻排序规则。
- `ContentView.swift` 只展示 `GameState` / `GameModels` 已给出的字段，并转发点选聚焦事件。
- `GameStateTests.swift` 和 `RulesSmokeTest.swift` 已有反制建议场景，可复用 `enemyThreatIntentScenario(axisSpent: true)` 和 smoke 中的 `enemyThreatGame`。

## 实现步骤

1. 在 `GameModels.swift` 新增轻量模型，例如 `EnemyThreatCountermeasurePriorityFactor` 和 `EnemyThreatCountermeasureComparisonPreview`，或在 `EnemyThreatCountermeasurePreview` 上新增 computed properties。
   - `priorityFactors` 至少包含：可执行、击毁、优先值、路线、执行单位、目标、坐标等现有排序维度中的有效项。
   - `prioritySummary` 用中文短句汇总本建议排序依据。
   - 模型不得依赖 SwiftUI，不重新计算战斗、路线、整补或 AI 结果。
2. 在 `GameState.swift` 新增只读 helper，例如 `enemyThreatCountermeasureComparisonPreviews(for:limit:)`。
   - 输入可以是已排序建议数组，也可默认使用 `visibleEnemyThreatCountermeasurePreviews`。
   - 输出前 N 条建议之间的相邻对比，说明左侧建议领先右侧建议的第一个关键排序差异。
   - 对比必须复用 `enemyThreatCountermeasureSort` 对应的维度顺序，不能发明新排序。
   - 不写入 `@Published` 状态，不调用真实移动、攻击、整补、部署、战术命令或 AI 方法。
3. 在 `ContentView.swift` 的 `EnemyThreatCountermeasurePanel` / `EnemyThreatCountermeasureRow` 中展示：
   - 每条建议的 `prioritySummary` 或前 2-3 个 `priorityFactors`。
   - 列表顶部或首条附近展示“领先原因/对比”提示，例如“首选：可击毁威胁来源，优先值高于撤退”。
   - 无障碍 label 应包含排序/对比解释。
   - UI 只消费模型/状态层字段，不自行比较 score 或路线。
4. 在 `GameStateTests.swift` 增加断言：
   - 反制建议存在非空 `priorityFactors` 和 `prioritySummary`。
   - 前两条建议的对比预览非空，且 `leading.id == countermeasures[0].id`、`trailing.id == countermeasures[1].id`。
   - 对比原因对应现有排序维度，且不会改变 active faction、单位、地格、战报、消息、latest result。
   - `limit: 0` 返回空，`limit: 1` 不生成相邻对比。
5. 在 `RulesSmokeTest.swift` 增加同等轻量断言，覆盖排序解释、相邻对比和只读无副作用。
6. 更新文档：
   - `WW2Tactics/README.md`：已实现列表和战斗辅助段落加入排序/对比解释。
   - `md/flow/flow.md`：GameModels、GameState、敌方意图反制建议和测试映射加入排序对比解释。
   - `md/flow/flowchart.md`：核心逻辑图和规则状态图加入排序/对比解释节点。
   - `md/test/test.md`：当前 smoke / stage regression 基线加入反制建议排序对比解释。
   - `update_log.md`：追加 v1.16 记录，先写本地轻量检查占位，Agent C 完成云端验收后再补 run/artifact。

## 验收标准

1. 反制建议列表能显示每条建议的排序依据，首条与次条能显示相邻对比原因。
2. 排序/对比解释只从已有排序字段派生，不改变建议顺序、score、执行入口或规则状态。
3. XCTest 和 smoke test 覆盖排序解释、相邻对比、limit 边界和只读无副作用。
4. 本地轻量检查通过。
5. Agent B 在 `main` 提交并 `git push origin main`。
6. GitHub Actions 生成未加密 artifact，Agent C 下载并核对 `ci-artifact-manifest.json`、`ci-failure-summary.md`、`junit.xml`、`rules-smoke.log`、`xcodebuild.log` 和 `WW2Tactics.xcresult`。
7. Agent C 确认 artifact manifest 的 `branch=main`、`commitSha`、`runId`、`runAttempt` 与最新 `origin/main` 完全一致。

## 风险和禁止项

- 风险：如果 UI 自行比较 score，后续排序规则变更会让展示失真；必须把对比逻辑放在 `GameState` 或模型纯派生层。
- 风险：排序解释文字过长可能挤压侧栏；展示应紧凑，正文可换行，按钮文本必须在窄屏不溢出。
- 风险：只读 helper 若误调用真实执行方法会污染状态；测试必须锁住 commandPoints、units、tiles、battleLog、message 和 latest result。
- 禁止绕过 `GameState` 在 View 中实现核心规则或排序规则。
- 禁止改动战斗数值、AI、移动、整补、胜负或执行桥接语义。
- 禁止伪造本地测试、云端 run 或 artifact 验收。
