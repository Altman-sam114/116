# v1.44 据点防守复核细分

## 版本分配依据

`md/prompt/v1（地图操作体验）` 已完成到 v1.43。v1.43 增强了战场态势响应历史查看，本轮继续沿着敌方威胁反制闭环推进，编号递增为 v1.44。

## 背景

当前据点防守反制已经能给出进驻或封堵建议，并在敌方回合后发布复核。但复核内容仍偏粗，只显示据点归属、防守单位和 AI 总览，无法明确说明防守动作到底是进驻还是封堵、防守单位是否仍在防守位、威胁源是否仍压迫据点。

## 目标

- 在敌方回合结束后的 `.objectiveDefense` 复核中细分防守动作、守点位置、威胁来源和据点结果。
- 复核必须基于真实执行后的 baseline 和真实 AI 后状态，不模拟“如果没防守会怎样”。
- UI 只读展示结构化复核细节，定位按钮仍只定位目标，不执行命令。
- 补充 XCTest 和 smoke test，覆盖据点防守复核新增字段和 comparison 顺序。
- 同步 README、flow、test、prompt README 和 update_log。

## 非目标

- 不调整 objectiveDefense 建议排序、评分、AI 行为或移动攻击规则。
- 不新增第三方依赖、资产管线或新的全局 `@Published` 状态。
- 不构造不稳定的 AI 失败场景作为 smoke 基线。
- 不把规则判断写进 SwiftUI View。

## 架构依据

- `GameModels.swift` 放置复核结构、枚举和派生展示字段。
- `GameState.swift` 在 `countermeasureFollowUpOutcome` 中根据 baseline 和 AI 后状态生成复核。
- `ContentView.swift` 只读取 `EnemyThreatCountermeasureFollowUpSummary` 展示，不重算规则。
- `GameStateTests.swift` 与 `RulesSmokeTest.swift` 复用现有 `enemyThreatFollowUpScenario(axisSpent:)`。

## 实现步骤

1. 新增 `EnemyThreatObjectiveDefenseFollowUpDetail` 和结果枚举，记录计划动作、据点归属、防守单位、威胁源与标题摘要。
2. 扩展 `EnemyThreatCountermeasureFollowUpSummary`，为 `.objectiveDefense` 增加可选 detail，并让 outcome level 优先读取结构化结果。
3. 在 `GameState.countermeasureFollowUpOutcome` 的 `.objectiveDefense` 分支中追加“守点位置”“威胁来源” comparisons，并返回 detail。
4. 在复核卡片中展示 detail 的标题和摘要，保持定位入口只读。
5. 更新 XCTest 和 smoke test，断言新增 comparison、detail、结论和态势响应摘要。
6. 更新文档和日志。

## 验收标准

- 据点防守复核 comparisons 至少包含“据点归属”“守点位置”“威胁来源”“防守单位”“AI总览”。
- 复核 detail 能区分进驻/封堵、守点是否成功、据点是否失守、防守单位与威胁源状态。
- `battlefieldSituationResponseSummary` 仍优先展示复核，并以据点归属作为 leading result。
- UI 展示新增 detail，但不新增可执行规则入口。
- 本地轻量检查通过，云端最新 artifact 与 `origin/main` 最新 commit 完全一致。

## 风险

- 中文文案断言过多会导致测试脆弱，应优先断 enum、字段、坐标和少量关键结果。
- 如果 AI 场景发生变化，objectiveDefense 成功基线可能需要同步调整。
- outcome level 从结构化 detail 派生后，要保证非 objectiveDefense 路径保持原行为。

## 禁止项

- 禁止绕过 `GameState` 在 View 中实现规则。
- 禁止调整 AI 决策或 objectiveDefense 建议生成逻辑。
- 禁止把本地输出或旧 artifact 冒充本轮云端结果。
