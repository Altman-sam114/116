# v1.3（战术命令结果摘要与战报强化）

## 版本号

v1.3

## 版本分配依据

当前 `md/prompt/v1（地图操作体验）/` 已完成 `v1.0` 路线与战斗预判、`v1.1` 火力风险与安全接敌、`v1.2` 普通攻击战斗结果回放。`v1.2` 明确遗留“火炮弹幕、突破突击等战术命令尚未进入结果摘要”，本轮继续围绕地图操作后的反馈闭环，递增为 `v1.3`。

## 背景

当前普通攻击执行后，`GameState.latestCombatResult` 会驱动侧栏战斗结果面板，展示双方 HP 前后、实际反击、击毁、追击、经验、晋升和士气变化。但火炮弹幕和突破突击仍只依赖 `message` / `battleLog`，玩家无法在同一块结果区域看到指令点消耗、无反击、压制状态、防御姿态消耗和 HP/士气前后对比。EasyTech 风格战棋中，战术命令是关键操作反馈，执行后也应有结构化战果回放。

## 目标

- 在 `GameModels.swift` 新增 `TacticalCommandResultSummary`，复用 `CombatantResultSnapshot` 记录施放者和目标的 HP、经验、士气、军衔前后状态。
- 在 `GameState.swift` 新增 `@Published private(set) var latestTacticalCommandResult: TacticalCommandResultSummary?`。
- 在 `useTacticalCommand(_:casterID:targetID:)` 中从真实结算前后生成结构化摘要，覆盖火炮弹幕和突破突击。
- 摘要必须记录：命令类型、施放者、目标、伤害、指令点消耗、士气损失、状态效果、目标是否击毁、目标防御姿态是否被消耗、是否无反击。
- 普通攻击和战术命令的“最新结果”在 UI 上应互斥展示：普通攻击写入 `latestCombatResult` 时清理 `latestTacticalCommandResult`；战术命令写入 `latestTacticalCommandResult` 时清理 `latestCombatResult`。
- 失败的战术命令不得生成新的战术命令结果摘要。
- 重开和切换战役时清理普通攻击结果和战术命令结果。
- 在 `ContentView.swift` 的侧栏战报前增加战术命令结果面板，复用现有 `CombatantResultLine`、`CombatResultMetric`、`CombatResultDetailRow` 的视觉语言，显示命令名、BRG/BRK、伤害、指令点消耗、士气损失、状态效果、无反击、防御姿态消耗、击毁和双方 HP 前后。
- 强化战术命令战报文本，使日志中包含 HP 前后、指令点消耗、士气/状态、无反击、防御姿态消耗等高信号信息。
- 补充 XCTest 和规则 smoke test，覆盖火炮弹幕、突破突击、目标防御姿态消耗、失败命令不生成摘要、普通攻击/战术命令结果互斥、AI 战术命令也写入摘要。
- 同步更新 `WW2Tactics/README.md`、`md/flow/flow.md`、`md/flow/flowchart.md` 和 `update_log.md`。

## 非目标

- 不改变火炮弹幕、突破突击的伤害公式、指令点消耗、射程、状态效果或 AI 选择逻辑。
- 不新增动画、音效、图片、美术资源或新资产管线。
- 不把结果摘要写成只能 UI 展示、无法测试的状态。
- 不让 View 解析 `battleLog` 字符串或承担规则推导。
- 不重构完整战报系统，不新增持久战报数据表。
- 不扩大战役、科技、生产队列、存档或 AI 攻防策略。

## 当前架构依据

- `GameModels.swift` 已有 `TacticalCommand`、`TacticalCommandPreview`、`CombatantResultSnapshot` 和 `CombatResultSummary`。战术命令执行结果应新增独立模型，不复用预览类型表示已执行结果。
- `GameState.swift` 是规则状态机。战术命令执行点是 `useTacticalCommand(_:casterID:targetID:)`；伤害公式在 `tacticalCommandDamage(command:caster:target:)`；AI 也通过 `useTacticalCommand` 执行命令。
- `ContentView.swift` 只能读取 `GameState` 的结构化结果并展示，不能从日志文本反推命令结果。
- `GameStateTests.swift` 和 `Tools/RulesSmokeTest.swift` 已有战术命令基础行为测试，是本轮应扩展的主要入口。

## 实现步骤

1. 在 `GameModels.swift` 的 `TacticalCommandPreview` 附近新增 `TacticalCommandResultSummary`：
   - 字段建议包括 `id`、`command`、`caster`、`target`、`damage`、`commandCost`、`moraleDamage`、`statusEffect`、`didDestroyTarget`、`didConsumeTargetEntrenchment`、`didAvoidCounterAttack`。
   - `caster` / `target` 使用 `CombatantResultSnapshot`。
   - 可添加派生属性：`targetHPDelta`、`targetMoraleDelta`、`didPromoteCaster` 等若 UI 需要。
2. 在 `GameState.swift`：
   - 新增 `latestTacticalCommandResult`。
   - `loadScenario` 清理该状态。
   - 普通攻击成功写入 `latestCombatResult` 时清理 `latestTacticalCommandResult`。
   - 战术命令成功写入 `latestTacticalCommandResult` 时清理 `latestCombatResult`。
   - 失败命令只更新 `message`，不得清理或覆盖已有结果，除非实现明确记录并有测试；推荐失败不生成新摘要。
3. 在 `useTacticalCommand` 内：
   - 结算前保存 `caster`、`target` 起始状态和 `target.isEntrenched`。
   - 按现有顺序扣指令点、扣 HP、清防御姿态、设置施放者行动状态、授予经验、降低士气、设置战术状态、更新据点和胜负。
   - 结算后读取最终 `caster` / `target`，生成 `TacticalCommandResultSummary`。
   - 战报文本补充 HP 前后、士气、状态、无反击、指令点消耗、防御姿态消耗和击毁信息。
4. 在 `ContentView.swift`：
   - `InspectorPanel` 在战报前优先显示普通攻击结果；若没有普通攻击结果但有 `latestTacticalCommandResult`，显示战术命令结果。
   - 新增 `TacticalCommandResultSummaryView`，复用现有小组件，保持紧凑、可读、Dynamic Type，不能遮挡地图。
   - 使用 `TacticalCommand` 的图标和 accent color，不新增手写 SVG。
5. 在 `GameStateTests.swift`：
   - 扩展 `testArtilleryBarrageUsesCommandPointsAndSuppressesTarget`：断言摘要字段、HP/经验/士气前后、指令点、状态、无反击、普通攻击结果被清理。
   - 扩展 `testBreakthroughAssaultUsesCommandPointsAndAvoidsCounterattack`：断言突破摘要字段、伤害高于普通攻击但无反击、状态和日志。
   - 扩展 `testArtilleryBarrageConsumesEntrenchedTargetDefense`：断言 `didConsumeTargetEntrenchment` 和日志包含防御姿态消耗。
   - 扩展 `testTacticalCommandRequiresEnoughCommandPoints`：失败命令不生成新摘要、不扣点、不伤害。
   - 增加或扩展 AI 战术命令测试：`endTurn()` 后若 AI 使用火炮弹幕，`latestTacticalCommandResult` 指向轴心命令结果。
6. 在 `Tools/RulesSmokeTest.swift`：
   - 在火炮弹幕和突破突击 smoke 链路中断言 `latestTacticalCommandResult` 的命令、伤害、HP 前后、指令点、士气、状态、无反击、防御姿态消耗。
   - 保留原有普通攻击 `latestCombatResult` smoke 断言。
7. 更新文档：
   - README 已实现列表加入战术命令结果摘要和强化战报。
   - flow/flowchart 补充战术命令结果摘要与 UI 展示链路。
   - update_log 追加 v1.3 记录，先记录本地检查，云端验收由 Agent C 补齐或确认。

## 文档更新要求

- 用户可见行为变化必须更新 `WW2Tactics/README.md`。
- 核心规则状态变化必须更新 `md/flow/flow.md` 和 `md/flow/flowchart.md`。
- 测试策略不变时无需修改 `md/test/test.md`。
- 追加 `update_log.md` 的 v1.3 记录，记录本地轻量检查、commit、push 和 Agent C 云端验收事实。

## 验收标准

- 火炮弹幕和突破突击成功执行后，`latestTacticalCommandResult` 准确反映命令、施放者/目标前后状态、伤害、指令点消耗、士气损失、状态效果、无反击、防御姿态消耗和击毁结果。
- 普通攻击与战术命令结果在 UI 上互斥展示，不显示过期的另一类结果。
- 失败战术命令不会生成虚假的最新结果。
- AI 使用战术命令时也能写入结构化结果并保持战报一致。
- UI 侧栏能显示战术命令结果，不遮挡地图，不把规则写进 View。
- XCTest 和 smoke test 覆盖新增摘要行为。
- 本地轻量检查通过。
- Agent B 在 `main` 上提交并推送到 `origin/main`。
- Agent C 下载最新 run 的 artifact，核对 manifest 的 `branch=main`、`commitSha`、`runId`、`runAttempt` 与 `origin/main` 一致，并确认静态检查、规则 smoke 和 Xcode build-for-testing 通过。

## 风险和禁止项

- 禁止改变战术命令数值、AI 决策或普通攻击规则。
- 禁止通过解析 `battleLog` 文本生成结果摘要。
- 禁止让 View 执行规则判定或绕过 `GameState`。
- 禁止让预览、聚焦、快速定位或失败命令产生结果摘要。
- 禁止扩大到动画、音效、完整战报持久化或新资源。
- 禁止创建长期分支、PR 或推送非 `main`。
- 禁止在没有真实云端 artifact 核对时宣称 Agent C 验收通过。
