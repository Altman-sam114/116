# v1.20 AI回合行动时间线

## 版本号

v1.20

## 版本分配依据

当前 `md/prompt/v1（地图操作体验）/` 最新功能版本为 v1.19（反制建议敌方回合复核）。v1.19 已经能在敌方回合后给出保守复核，但仍只能看到聚合 AI 摘要和复核结论，缺少“敌方这一回合按什么顺序做了哪些动作”的可读记录。本轮继续推进敌方回合反馈链，因此递增为 v1.20。

## 背景

EasyTech 风格战棋的敌方回合不只是结算数字，玩家需要理解敌军行动顺序、关键攻击、推进、占点和后勤动作。当前 `AIPhaseSummary` 只有计数和总伤害；v1.19 的复核也只能基于 HP、位置、据点归属和威胁源存活进行保守判断。如果没有逐行动时间线，玩家无法快速知道“哪支敌军移动到哪里”“谁攻击了谁”“谁部署/整补/占点”。本轮应在不改变 AI 决策和数值的前提下记录真实成功动作，展示最近一次敌方回合时间线。

## 目标

1. 新增 AI 回合逐行动事件模型，记录真实成功的敌方整补、部署、战术命令、攻击、移动和占点结果。
2. `GameState` 在现有 AI 成功路径中追加只读事件，并在 `finishAIPhaseRecording()` 后发布到最新 `AIPhaseSummary` 或独立 `latestAIPhaseTimeline`。
3. 事件至少包含：
   - 顺序号、阵营、回合、动作类型。
   - 行动单位名称、目标名称、起点/终点坐标。
   - 关键数值：伤害、恢复、部署兵种、指令点消耗、据点名、是否击毁或占点。
   - 简短 `summary` 和用于 UI 的短码。
4. 侧栏在 AI 回合摘要附近展示最近一次敌方回合时间线，列出前若干条动作，保持紧凑、可换行并支持 VoiceOver。
5. 地图可低风险显示最近一次 AI 动作的短码标记；若接入会造成 UI 复杂或测试不稳，本轮只做侧栏时间线。
6. 补充 XCTest 和规则 smoke test，覆盖战术命令、移动+攻击、整补、部署、占点/机动追击，以及重开/切战役清理。
7. 同步 README、flow、flowchart、test 和 update_log。

## 非目标

- 不改变 AI 决策顺序、目标选择、攻击数值、移动规则、整补成本、部署逻辑、胜负条件或指令点收入。
- 不新增动画、音效、逐帧回放、暂停/播放控制、镜头自动移动或真实美术资产。
- 不把 AI 事件记录做成通用事件总线；本轮只服务最近一次 AI 回合摘要和侧栏展示。
- 不让 `ContentView` 推断战斗、移动、占点或后勤结果。
- 不新增第三方库、网络依赖或复杂资产管线。
- 不重构当前 `ObservableObject/@Published` 架构。

## 当前架构依据

- `GameModels.swift` 已有 `AIPhaseSummary` 聚合摘要，以及普通攻击、战术命令、部署、整补、占点等结果摘要模型；本轮应在这里新增轻量 `AIPhaseTimelineEvent` 类型。
- `GameState.swift` 已有 `beginAIPhaseRecording(for:)`、`recordAIPhaseAction(_:)`、`finishAIPhaseRecording()` 和所有 AI 成功路径；本轮应在规则成功路径内记录事件，不能由 View 事后推断。
- `ContentView.swift` 已在结果区显示 `AIPhaseSummaryView`，本轮应在附近展示 timeline，沿用现有紧凑卡片风格。
- `GameStateTests.swift` 和 `RulesSmokeTest.swift` 已有 `axisTacticalCommandScenario`、`axisFullAdvanceScenario`、`axisDeploymentResultScenario`、`axisReinforcementResultScenario` 和 `axisManeuverPursuitScenario`，可复用作为稳定测试场景。

## 实现步骤

1. 在 `GameModels.swift` 新增：
   - `AIPhaseTimelineEventKind`，建议覆盖 `reinforcement`、`deployment`、`tacticalCommand`、`attack`、`move`、`objectiveCapture`。
   - `AIPhaseTimelineEvent`，字段建议为 `order`、`faction`、`turn`、`kind`、`actorName`、`targetName`、`from`、`to`、`value`、`detail`、`didDestroyTarget`、`didCaptureObjective`，并提供 `shortCode`、`summary`。
   - 可选在 `AIPhaseSummary` 中加入 `timeline: [AIPhaseTimelineEvent]`；若为避免扩大 summary 初始化影响，也可新增 `@Published private(set) var latestAIPhaseTimeline: [AIPhaseTimelineEvent]`。
2. 在 `GameState.swift` 新增私有 AI timeline 缓冲：
   - `activeAIPhaseTimeline: [AIPhaseTimelineEvent]`
   - `recordAIPhaseTimelineEvent(...)`，仅当 `activeAIPhaseBaseline` 存在且 `baseline.faction == activeFaction` 时记录。
3. 在现有成功路径记录事件：
   - `reinforce(unitID:)` 成功后记录整补事件，包含单位名、恢复量、坐标和指令点消耗。
   - `deploy(kind:at:)` 成功后记录部署事件，包含新单位名、兵种、坐标和消耗。
   - `useTacticalCommand` 成功后记录战术命令事件，包含施放者、目标、伤害、命令名、击毁和指令点消耗。
   - `attack(attackerID:targetID:)` 成功后记录攻击事件，包含攻击者、目标、伤害、反击、击毁。
   - `move(unitID:to:)` 成功后记录移动事件，包含单位、起点、终点；若 `updateObjectiveControl()` 导致占点，可记录独立占点事件或在移动事件 detail 中标注。
   - `applyObjectiveCaptureReward` 成功后记录占点事件，包含据点名、坐标、原归属、新归属和奖励。
4. 确保记录顺序与真实执行顺序一致：
   - 每条事件 `order` 从 1 递增。
   - 事件数量应等于或不少于 `AIPhaseSummary.totalActions`，因为占点可能作为额外结果事件。
   - 被动据点休整、补给恢复、预览和失败动作不记录 timeline 事件。
5. 清理规则：
   - `beginAIPhaseRecording` 开始时清空 active timeline。
   - `finishAIPhaseRecording` 发布最新 timeline。
   - `loadScenario()`、重开、切战役清理 timeline。
   - 玩家预览、失败部署、普通聚焦不生成 timeline。
6. `ContentView.swift`：
   - 在 AI 摘要卡中或其后展示“行动时间线”，最多显示 5 条，超出时显示“另有 N 条”。
   - 每条事件显示顺序、短码、行动单位、目标/坐标和 summary。
   - 支持 VoiceOver，不能只靠颜色区分动作类型。
7. 测试：
   - XCTest 覆盖 tactical command timeline、move+attack timeline、deployment/reinforcement timeline、objective capture timeline、scenario reset 清理、failed/player action 不生成。
   - Smoke test 复用现有 AI 摘要 smoke 场景，断言 timeline 存在、顺序号稳定、kind 与 summary 计数匹配、关键 actor/target 名称存在。
8. 文档更新：
   - `WW2Tactics/README.md`：AI 和战斗辅助加入“AI 回合行动时间线”。
   - `md/flow/flow.md`：GameModels、GameState、AI 回合和测试映射加入 timeline。
   - `md/flow/flowchart.md`：AI summary 后加入 timeline 节点。
   - `md/test/test.md`：当前 smoke / Stage Regression 基线加入 timeline。
   - `update_log.md`：追加 v1.20 记录，本地验证后写具体命令结果，Agent C 验收后补云端结果。

## 验收标准

1. 结束玩家回合后，侧栏除 AI 聚合摘要外还能看到敌方回合行动时间线。
2. 时间线来自 `GameState` 真实成功路径，至少覆盖整补、部署、战术命令、攻击、移动和占点事件。
3. 事件顺序、动作类型、行动单位、目标/坐标和关键数值可测试，且与 `AIPhaseSummary` 聚合计数不矛盾。
4. 玩家预览、失败行动、普通聚焦和被动休整不生成 timeline 事件。
5. 重开、切战役和下一次 AI 回合会清理或覆盖旧 timeline。
6. 不改变 AI、战斗、移动、整补、部署、战术命令、胜负和反制建议语义。
7. 本地轻量检查通过。
8. Agent B 在 `main` 提交并 `git push origin main`。
9. GitHub Actions 生成未加密 artifact，Agent C 下载并核对 `ci-artifact-manifest.json`、`ci-failure-summary.md`、`junit.xml`、`rules-smoke.log`、`xcodebuild.log` 和 `WW2Tactics.xcresult`。
10. Agent C 确认 artifact manifest 的 `branch=main`、`commitSha`、`runId`、`runAttempt` 与最新 `origin/main` 完全一致。

## 风险和禁止项

- 风险：`attack`、`move`、`deploy`、`reinforce`、`useTacticalCommand` 也会被玩家路径调用，timeline helper 必须只在 active AI phase 中记录。
- 风险：占点可能由移动或机动追击触发，应避免重复计数到 `AIPhaseSummary.totalActions`；占点可以作为结果事件，但不应改变动作计数。
- 风险：AI 回合可能因胜负提前结束，timeline 必须保留已成功执行的事件，并容忍后续 reset 未发生。
- 风险：时间线过长会挤压侧栏，应默认截断展示并保留完整数据供测试。
- 禁止绕过 `GameState` 在 View 中推断 AI 行动。
- 禁止改变 AI 决策、数值、移动、整补、部署、战术命令、胜负或反制建议语义。
- 禁止伪造本地测试、云端 run 或 artifact 验收。
