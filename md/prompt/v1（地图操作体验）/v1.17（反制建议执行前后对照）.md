# v1.17 反制建议执行前后对照

## 版本号

v1.17

## 版本分配依据

当前 `md/prompt/v1（地图操作体验）/` 最新功能版本为 v1.16（反制建议排序对比解释）。本轮继续围绕 v1 地图操作体验阶段推进反制建议可读性和可玩性，因此递增为 v1.17。

## 背景

v1.10 到 v1.16 已经形成敌方意图、反制建议、点选聚焦、地图标记、执行入口桥接、收益解释和排序对比解释。玩家现在知道“为什么推荐这条”，但还缺少执行前后的直观后果对照：当前威胁会造成什么结果，采纳该建议后预计能改善什么。EasyTech 风格参谋提示需要让玩家快速判断“做不做这一手”，而不是只看收益标签或排序分。

## 目标

1. 为 `EnemyThreatCountermeasurePreview` 派生只读执行前后对照，展示当前威胁后果、采纳建议后的预计结果和净改善。
2. 对四类反制建议分别给出可读对照：
   - 抢先打击：当前威胁伤害 vs 先手打击后威胁来源剩余/击毁。
   - 撤出危险区：当前威胁后目标 HP vs 撤退后预计 HP。
   - 据点防守：当前据点被夺风险 vs 进驻/封堵后的守点动作。
   - 整补支撑：当前威胁后目标 HP vs 整补后承受威胁的预计 HP。
3. 在反制建议面板中展示紧凑“前/后/改善”指标，并把对照摘要加入无障碍文本。
4. 补充 XCTest 和规则 smoke test，覆盖四类建议的执行前后对照、limit/只读边界和现有建议生成不变。
5. 同步 README、flow、flowchart、test 和 update_log。

## 非目标

- 不改变反制建议排序、score 公式、敌方意图预判、AI 行动或战斗数值。
- 不让反制建议直接执行攻击、移动、整补或战术命令。
- 不把核心规则或对照计算写入 `ContentView`。
- 不新增第三方库、网络依赖、图片资产或复杂资产管线。
- 不重构项目架构，不迁移 `ObservableObject/@Published`。

## 当前架构依据

- `GameModels.swift` 已定义 `EnemyThreatCountermeasurePreview`、`benefitMetrics`、`priorityFactors` 和执行入口模型；执行前后对照应作为纯派生模型或 computed property 放在这里，避免 UI 计算规则。
- `GameState.swift` 已在创建反制建议时保存 `threatKind`、`projectedDamage`、`projectedFriendlyHPAfterAction`、`projectedRecoveredHP`、`willDestroyEnemy`、`destination`、`routeCost` 和 `score` 等字段；本轮优先复用这些字段，只有必要时才在状态层补只读 helper。
- `ContentView.swift` 只展示模型/状态层给出的对照字段，并继续只调用 `focusEnemyThreatCountermeasure(_:)`。
- `GameStateTests.swift` 和 `RulesSmokeTest.swift` 已有 `enemyThreatIntentScenario(axisSpent: true)`，可复用生成四类反制建议。

## 实现步骤

1. 在 `GameModels.swift` 新增轻量模型，例如 `EnemyThreatCountermeasureImpactMetric` 和 `EnemyThreatCountermeasureImpactPreview`。
   - 字段建议包含 `beforeTitle`/`beforeValue`、`afterTitle`/`afterValue`、`deltaTitle`/`deltaValue`、`summary`。
   - `EnemyThreatCountermeasurePreview` 增加 `impactPreview` 和 `impactSummary` computed properties。
   - 对照只读派生自已有字段，不调用战斗、移动、整补或 AI 方法，不 import SwiftUI。
2. 四类建议对照要求：
   - `.firstStrike`：before 可显示“威胁 -N”或威胁类型，after 显示“击毁”或“敌 HP X”，delta 显示“解除/削弱威胁”。
   - `.withdraw`：before 使用 `threatTargetProjectedHPAfterDamage` 类似字段；若当前模型没有该字段，可在构造建议时补充一个只读 `projectedThreatTargetHPBeforeAction`，不要重新计算 View 逻辑。
   - `.objectiveDefense`：before 显示“据点被夺风险”，after 显示“进驻/封堵 q,r”，delta 显示“守点/封堵”。
   - `.reinforce`：before 使用威胁后 HP，after 使用整补后再承受威胁的预计 HP；若当前模型只保存整补后 HP，允许在状态层构造时补充预计承受威胁后 HP。
3. 若为了撤退/整补对照需要新增字段，应只扩展 `EnemyThreatCountermeasurePreview` 构造参数，并在所有构造点和过期建议测试中补齐。
4. 在 `ContentView.swift` 的 `EnemyThreatCountermeasureRow` 中展示对照指标：
   - 使用已有 `CombatResultMetric` 或等价紧凑样式展示“当前 / 采纳 / 改善”。
   - 文本允许换行，不能挤压按钮主体或只用颜色表达差异。
   - 无障碍 label 应包含 `impactSummary`。
5. 在 `GameStateTests.swift` 中扩展反制建议测试：
   - 四类建议都有非空 `impactPreview.summary` / `impactSummary`。
   - 撤退和整补的“采纳后” HP 高于当前威胁后 HP。
   - 抢先打击击毁或削弱威胁来源；据点防守显示守点/封堵动作。
   - 生成对照不改变 active faction、command points、units、tiles、battleLog、message 和 latest result。
6. 在 `RulesSmokeTest.swift` 加同等轻量断言，覆盖四类建议和只读无副作用。
7. 更新文档：
   - `WW2Tactics/README.md`：已实现、地图交互/战斗辅助和 smoke 覆盖描述加入执行前后对照。
   - `md/flow/flow.md`：GameModels、GameState、3.13 反制建议和测试映射加入执行前后对照。
   - `md/flow/flowchart.md`：核心逻辑图和规则状态图加入 impact/前后对照节点。
   - `md/test/test.md`：当前 smoke 和 Stage Regression 基线加入反制建议执行前后对照。
   - `update_log.md`：追加 v1.17 记录，本地验证后写具体命令结果，Agent C 验收后补云端结果。

## 验收标准

1. 反制建议行能显示执行前后对照，四类建议都有明确“当前/采纳/改善”摘要。
2. 对照只读派生，不改变建议排序、执行入口、AI、移动、战斗或整补规则。
3. XCTest 和 smoke test 覆盖四类建议、HP 改善/守点动作、只读无副作用。
4. 本地轻量检查通过。
5. Agent B 在 `main` 提交并 `git push origin main`。
6. GitHub Actions 生成未加密 artifact，Agent C 下载并核对 `ci-artifact-manifest.json`、`ci-failure-summary.md`、`junit.xml`、`rules-smoke.log`、`xcodebuild.log` 和 `WW2Tactics.xcresult`。
7. Agent C 确认 artifact manifest 的 `branch=main`、`commitSha`、`runId`、`runAttempt` 与最新 `origin/main` 完全一致。

## main 直推和云端验证要求

- 当前固定使用 `main` 作为唯一上传、提交、推送和云端验证分支。
- 本轮不使用 `smalldata_test`、`develop`、`codeb/...` 或 PR 流程。
- Agent B 开始前必须执行：

```sh
git fetch origin
git switch main
git pull --ff-only origin main
git status --short --branch
```

- Agent B 完成本地轻量检查后，用 `v1.17: 增加反制建议执行前后对照` 提交并 `git push origin main`。
- 若仓库没有 `origin/main`、当前分支不是 `main`、工作区有无法判断归属的冲突，或没有 push/artifact 权限，必须停止并说明阻塞。

## 风险和禁止项

- 风险：当前模型若缺少“威胁后 HP”字段，不能让 View 现场重算；应在状态层构造建议时补只读字段。
- 风险：对照文字过长会挤压侧栏；UI 应优先使用短指标和可换行摘要。
- 风险：对照可能被误解为已执行结果；文案必须说明是预计/采纳后。
- 禁止绕过 `GameState` 在 View 中实现核心规则。
- 禁止改动战斗数值、AI、移动、整补、胜负、排序或执行桥接语义。
- 禁止伪造本地测试、云端 run 或 artifact 验收。
