# v1.19 反制建议敌方回合复核

## 版本号

v1.19

## 版本分配依据

当前 `md/prompt/v1（地图操作体验）/` 最新功能版本为 v1.18（反制建议执行回放）。v1.18 已经让玩家在采纳反制建议并通过既有 ATK/MOVE/整补入口成功执行后看到“预计 / 实际 / 结果”的即时回放，但回合结束后仍无法确认敌方 AI 阶段是否真的绕开、削弱或打穿了这次反制。本轮继续推进同一条“敌方意图 -> 反制建议 -> 执行回放 -> 敌方回合复核”体验链，因此递增为 v1.19。

## 背景

EasyTech 风格的大地图战棋需要让玩家知道参谋建议是否经受住敌方回合验证。当前 v1.18 在执行撤退、守点、抢先打击或整补后只展示即时结果；`endTurn()` 会清理 `latestEnemyThreatCountermeasureExecutionResult` 并运行轴心国 AI，玩家只能从普通 AI 摘要中自行推断建议是否有效。这个推断成本高，且 AI 摘要没有绑定“刚才执行的反制建议”。本轮应在不改变 AI 行为的前提下，给出只读的敌方回合后复核摘要。

## 目标

1. 新增最近一次反制建议敌方回合复核模型，记录玩家本回合最新反制建议执行回放在敌方 AI 阶段后的验证结果。
2. 在 `endTurn()` 中于清理 v1.18 即时回放之前保存复核基线；保留 `latestEnemyThreatCountermeasureExecutionResult` 被回合切换清理的现有语义。
3. 在 `runAxisAI()` 和 `finishAIPhaseRecording()` 完成后发布只读复核摘要，对比反制建议执行后与敌方回合后的关键状态：
   - 抢先打击：威胁来源是否被击毁、是否还能造成伤害、目标单位 HP 是否保住。
   - 撤出危险区：目标单位敌方回合后是否存活、位置是否仍避开原威胁直接攻击点、HP 变化。
   - 据点防守：目标据点归属是否保持、守点单位是否存活或是否阻止原占点威胁。
   - 整补支撑：整补单位敌方回合后是否存活、HP 是否比整补前基线更稳。
4. 在侧栏反制回放或 AI 摘要附近展示“敌方回合复核”卡，明确这是回合后验证，不是新建议或自动执行。
5. 补充 XCTest 和规则 smoke test，覆盖至少直接威胁、撤退、守点/据点和整补复核，以及清理边界。
6. 同步 README、flow、flowchart、test 和 update_log。

## 非目标

- 不改变 AI 决策、AI 行动顺序、攻击数值、移动规则、整补成本、胜负条件、威胁意图生成或反制建议排序。
- 不保留 `latestEnemyThreatCountermeasureExecutionResult` 跨过 `endTurn()`；本轮新增独立 follow-up 结果。
- 不新增一键执行反制建议按钮。
- 不让 `ContentView` 推断威胁、战斗、路线、据点或 AI 结果。
- 不新增动画、音效、图片资产、第三方库、网络依赖或复杂资产管线。
- 不重构当前 `ObservableObject/@Published` 架构。

## 当前架构依据

- `GameModels.swift` 已定义 `EnemyThreatCountermeasureExecutionResultSummary` 和 `AIPhaseSummary` 等只读结果摘要；本轮应新增轻量 follow-up 模型放在这里。
- `GameState.swift` 是核心状态机，已有 `latestEnemyThreatCountermeasureExecutionResult`、`endTurn()`、`beginAIPhaseRecording(for:)`、`runAxisAI()` 和 `finishAIPhaseRecording()`；本轮应在这里保存基线并生成复核结果。
- `ContentView.swift` 只能展示 `GameState` 暴露的复核字段，不能计算复核结论。
- `GameStateTests.swift` 和 `RulesSmokeTest.swift` 已覆盖 v1.18 执行回放、AI 摘要和敌方意图测试，可复用小场景构造稳定复核用例。

## 实现步骤

1. 在 `GameModels.swift` 新增模型：
   - `EnemyThreatCountermeasureFollowUpResultKind`，建议覆盖 `survival`、`enemyHP`、`objective`、`position`、`aiImpact`、`recovery`。
   - `EnemyThreatCountermeasureFollowUpComparison`，字段建议为 `kind`、`title`、`beforeEnemyPhase`、`afterEnemyPhase`、`result`。
   - `EnemyThreatCountermeasureFollowUpSummary`，字段建议包含反制建议 ID/类型、执行入口、目标单位、威胁来源、目标名、坐标、`aiPhaseSummaryID` 或回合号、摘要标题、结论文本和对照数组，并提供 `detailSummary`。
2. 在 `GameState.swift` 新增：
   - `@Published private(set) var latestEnemyThreatCountermeasureFollowUpResult: EnemyThreatCountermeasureFollowUpSummary?`
   - 私有基线结构，例如 `PendingEnemyThreatCountermeasureFollowUpBaseline`，保存 v1.18 执行回放、执行后单位 HP/位置、威胁来源 HP/位置、目标据点归属和整补后 HP。
3. 在 `endTurn()` 中：
   - 在 `latestEnemyThreatCountermeasureExecutionResult = nil` 之前调用 helper 捕获基线。
   - 继续清理即时执行回放，保持 v1.18 回合切换语义。
   - 运行 `beginAIPhaseRecording`、`runAxisAI()`、`finishAIPhaseRecording()` 后，基于基线和当前真实状态发布 follow-up。
4. 复核生成规则：
   - 只基于已存在的真实状态和 `latestAIPhaseSummary`，不重新模拟 AI。
   - 如果目标单位、威胁来源或据点已不存在，应以“已被歼灭 / 已失守 / 已解除”形式写入结果，不崩溃。
   - 若没有待复核基线，`endTurn()` 不生成复核。
   - 新的反制建议执行回放发布时清理旧 follow-up，避免旧复核误导当前行动。
   - `loadScenario()`、重开、切战役清理 follow-up 和 pending baseline。
5. 在 `ContentView.swift` 的结果区展示 follow-up 卡：
   - 放在反制回放或 AI 回合摘要附近。
   - 展示反制类型、执行入口、目标、威胁来源、结论和前三项对照。
   - 文案使用“敌方回合复核”，不要暗示自动执行或新建议。
   - 保持现有紧凑侧栏风格，支持 VoiceOver。
6. 在 `GameStateTests.swift` 中补充测试：
   - 有 v1.18 执行回放时 `endTurn()` 清理即时回放，并生成独立 follow-up。
   - 直接威胁或抢先打击复核能反映威胁来源/目标单位敌方回合后状态。
   - 撤退复核能反映目标单位敌方回合后存活、位置和 HP 变化。
   - 据点防守复核能反映据点归属或占点威胁结果。
   - 整补复核能反映整补单位敌方回合后 HP/存活。
   - 没有反制建议执行回放时 `endTurn()` 不生成 follow-up；重开/切战役清理 follow-up。
7. 在 `RulesSmokeTest.swift` 中补充轻量断言，覆盖 follow-up 发布、即时回放清理、无基线不生成、重开/切战役清理。
8. 更新文档：
   - `WW2Tactics/README.md`：已实现能力、地图交互/战斗辅助和 smoke 覆盖加入敌方回合复核。
   - `md/flow/flow.md`：GameModels、GameState、AI 回合、反制建议和测试映射加入 follow-up。
   - `md/flow/flowchart.md`：核心逻辑图、回合/AI 流和反制建议流加入复核节点。
   - `md/test/test.md`：当前 smoke 和 Stage Regression 基线加入敌方回合复核。
   - `update_log.md`：追加 v1.19 记录，本地验证后写具体命令结果，Agent C 验收后补云端结果。

## 验收标准

1. 玩家执行一次反制建议后结束回合，v1.18 即时执行回放被清理，但侧栏出现独立的“敌方回合复核”。
2. 复核摘要至少能说明目标单位/威胁来源/据点/HP 在敌方 AI 阶段后的真实状态变化。
3. 无执行回放、普通聚焦、预览或未采纳建议的行动不会生成虚假 follow-up。
4. 重开、切战役和新的反制建议执行会清理旧 follow-up 或 pending baseline。
5. 不改变 AI、战斗、移动、整补、排序、胜负和执行桥接语义。
6. 本地轻量检查通过。
7. Agent B 在 `main` 提交并 `git push origin main`。
8. GitHub Actions 生成未加密 artifact，Agent C 下载并核对 `ci-artifact-manifest.json`、`ci-failure-summary.md`、`junit.xml`、`rules-smoke.log`、`xcodebuild.log` 和 `WW2Tactics.xcresult`。
9. Agent C 确认 artifact manifest 的 `branch=main`、`commitSha`、`runId`、`runAttempt` 与最新 `origin/main` 完全一致。

## 风险和禁止项

- 风险：敌方 AI 可能击毁目标单位、威胁来源或改变据点归属，复核必须容忍对象缺失并给出清晰结论。
- 风险：如果直接保留 v1.18 即时回放跨回合，会破坏既有清理语义；必须用独立 follow-up。
- 风险：复核容易被误读为新建议，UI 文案必须强调“敌方回合后验证”。
- 风险：测试场景中 AI 可能因胜负条件提前终止或行动路径变化，测试应使用稳定小场景并只断言关键不变量。
- 禁止绕过 `GameState` 在 View 中实现核心规则。
- 禁止改变 AI、战斗数值、移动、整补、胜负、排序或执行桥接语义。
- 禁止伪造本地测试、云端 run 或 artifact 验收。
