# v1.4（据点占领结果摘要与 CAP 反馈）

## 版本号

v1.4

## 版本分配依据

当前 `md/prompt/v1（地图操作体验）/` 已完成 v1.0 路线与战斗预判、v1.1 火力风险与安全接敌、v1.2 普通攻击结果回放、v1.3 战术命令结果摘要。本轮继续补齐地图行动后的反馈闭环，把据点占领也纳入结构化结果回放，递增为 v1.4。

## 背景

项目已有据点占领、占领奖励、OBJ 目标推进、据点条和胜负判定。当前真实占领/夺取据点时，只通过 `message` 和战报文字反馈；如果同一步触发胜利，`checkVictory()` 还会覆盖当前消息。玩家缺少像攻击和战术命令一样清晰的“本次占点带来了什么”的结果卡，也缺少地图上的最新占点短码反馈。

本轮目标是新增结构化据点占领结果摘要和 `CAP` 地图反馈，不改变任何占领数值或胜负规则。

## 目标

- 新增 `ObjectiveCaptureResultSummary`，记录最近一次真实占领/夺取据点的结果。
- `GameState` 发布 `latestObjectiveCaptureResult`，只在真实据点归属变化时写入。
- 摘要包含据点名、坐标、原归属、新归属、占领单位、指令点/士气/经验奖励、占领后据点进度。
- 普通攻击、战术命令和据点占领结果应避免互相显示过期内容；据点占领写入时清理普通攻击/战术命令结果。
- 重开、切换战役、普通攻击和战术命令应按需要清理占领结果，避免侧栏和地图显示旧 CAP。
- `ContentView` 在侧栏战报前显示据点占领结果卡；地图在最近占领据点显示 `CAP` 短码。
- 补充 XCTest 和规则 smoke test，覆盖占领敌方/中立据点摘要、奖励字段、进度字段、远距离 OBJ 推进不生成摘要、重开/切战役清理。

## 非目标

- 不修改占领奖励数值：指令 +3、士气 +8、经验 +10。
- 不修改据点归属、胜负、AI、补给、目标选择排序或 OBJ 快捷执行语义。
- 不新增动画、音效、美术资产、生产队列或新战役。
- 不把占领规则、奖励或进度计算写进 `ContentView`。
- 不改变现有 MOVE/ATK/POS/NEXT/OBJ/THR 执行入口和右键/点按语义。

## 当前架构依据

- `WW2Tactics/WW2Tactics/GameModels.swift` 放结构化数据类型；v1.2/v1.3 已有 `CombatResultSummary` 和 `TacticalCommandResultSummary` 可参考。
- `WW2Tactics/WW2Tactics/GameState.swift` 是规则状态机；`updateObjectiveControl()` / `applyObjectiveCaptureReward()` 是据点归属和奖励的唯一落点。
- `WW2Tactics/WW2Tactics/ContentView.swift` 只展示 `GameState` 输出；可复用现有结果卡视觉语言，不在 View 中重新计算规则。
- `WW2Tactics/WW2TacticsTests/GameStateTests.swift` 和 `WW2Tactics/Tools/RulesSmokeTest.swift` 锁定规则行为。

## 实现步骤

1. 在 `GameModels.swift` 新增 `ObjectiveCaptureResultSummary`：
   - 字段至少包含：`objectiveName`、`coordinate`、`capturingUnitName`、`capturingUnitKind`、`previousOwner`、`newOwner`、`commandPointReward`、`moraleReward`、`experienceReward`、`alliedScoreAfterCapture`、`axisScoreAfterCapture`、`totalObjectiveCount`。
   - 可提供 `actionTitle`、`ownerTransitionText`、`progressText` 等纯展示派生字段。
2. 在 `GameState.swift`：
   - 新增 `@Published private(set) var latestObjectiveCaptureResult: ObjectiveCaptureResultSummary?`。
   - `loadScenario()` 清理新结果。
   - `updateObjectiveControl()` 在归属变化后写入摘要。注意奖励后需要重新读取占领单位状态或使用常量记录奖励，避免拿到奖励前错误状态。
   - 据点占领写入摘要时清理 `latestCombatResult` 和 `latestTacticalCommandResult`。
   - 普通攻击和战术命令成功执行时清理 `latestObjectiveCaptureResult`，避免过期 CAP 与新的战斗结果并存。
   - 远距离 OBJ 推进只移动到中继格时不得生成摘要；只有 `scenario.tiles[index].owner` 真实变化才生成。
3. 在 `ContentView.swift`：
   - 在 `InspectorPanel` 的结果/战报区域加入 `ObjectiveCaptureResultSummaryView`，优先级建议：普通攻击结果 -> 战术命令结果 -> 据点占领结果 -> 战报。
   - 复用现有 `CombatResultMetric` / detail row 风格或保持一致的紧凑信息密度。
   - 在地图渲染链路传入最近占领坐标，仿现有引导目标标记增加 `CAP` 短码；只显示 `latestObjectiveCaptureResult.coordinate`。
   - 地图 accessibility 文案可包含“最新占领据点”。
4. 在 `GameStateTests.swift`：
   - 在已有据点奖励测试处追加摘要断言：敌方据点占领后摘要非空，包含据点、原归属 `.axis`、新归属 `.allies`、占领单位、奖励值和进度。
   - 新增/补充中立据点占领摘要断言。
   - 远距离 OBJ 推进到中继格不生成占领摘要，并保留 `guidedObjectiveCoordinate`。
   - `restart()` 和 `selectScenario(id:)` 清空摘要。
   - 普通攻击或战术命令执行后清空旧占领摘要。
5. 在 `RulesSmokeTest.swift`：
   - 在据点奖励段和 OBJ 直取段检查摘要存在、字段正确。
   - 在远距离 OBJ 推进段检查未生成摘要。
   - 在重开或普通聚焦/预览场景检查不会生成虚假摘要。
6. 更新文档：
   - `WW2Tactics/README.md`：已实现和战斗辅助加入据点占领结果卡、CAP 地图反馈。
   - `md/flow/flow.md`：据点占领流程补充结构化结果摘要。
   - `md/flow/flowchart.md`：状态写回和 UI 渲染补充占领结果。
   - `update_log.md`：追加 v1.4 记录，本地检查先写，云端验收由 Agent C 补齐。

## 文档更新要求

- 用户可见的占领结果卡和 CAP 地图反馈必须同步 README。
- 核心据点占领反馈逻辑变化必须同步 `md/flow/flow.md` 和 `md/flow/flowchart.md`。
- 测试命令、基线或策略若有变化才更新 `md/test/test.md`；本轮预计无需修改测试策略。
- `update_log.md` 必须记录 v1.4 核心变更、关键文件、本地验证、commit/push 和 Agent C 云端验收事实。

## 验收标准

- 真实占领/夺取据点后，`latestObjectiveCaptureResult` 字段完整且与实际据点归属、奖励和进度一致。
- 非占领移动、普通聚焦、预览、远距离 OBJ 中继推进不会生成虚假占领摘要。
- 侧栏显示据点占领结果卡，地图在最新占领据点显示 `CAP`。
- 普通攻击、战术命令和据点占领结果不会显示过期的另一类结果。
- README、flow、flowchart、update_log 与代码状态一致。
- 本地轻量检查通过。
- `main` 已 push，Agent C 已下载最新 `origin/main` run 的 artifact 并核对 manifest、summary、JUnit、规则 smoke、xcodebuild 日志和 `.xcresult`。

## 风险和禁止项

- 风险：`applyObjectiveCaptureReward()` 当前基于占领单位触发奖励，若摘要需要奖励后状态，必须重新读取 `scenario.units`。
- 风险：胜利消息可能覆盖占领消息，摘要不能依赖 `message`。
- 风险：地图 `CAP` 标记不能覆盖 OBJ 引导语义；二者可同时存在但必须可读。
- 禁止把占领规则写进 SwiftUI。
- 禁止修改 +3/+8/+10 奖励、胜负条件、AI 或目标排序。
- 禁止扩大到动画、音效、生产系统或美术资产。
- 禁止跳过 Agent C 对最新 `origin/main` artifact 的验收。
