# v1.42（据点防守取舍解释）

## 版本号

v1.42

## 版本分配依据

当前 `v1（地图操作体验）` 最新功能版本为 v1.41，已让战线态势响应卡能定位到地图响应标记。下一步应继续强化 EasyTech 风格战棋中“守点还是封堵”的战场决策可读性，解释据点防守反制建议为什么选择进驻据点或相邻封堵。因此本轮递增为 v1.42。

## 背景

现有 `EnemyThreatCountermeasurePreview` 已支持 `.objectiveDefense`，会在敌方据点占领威胁出现时选择己方单位进驻目标据点或移动到相邻格封堵，并已有收益、排序依据和执行前后预计对照。但 UI 目前只显示通用“守点/目的/路线”和 impact summary，玩家仍需要自己推断“为什么是进驻而不是封堵”。本轮要把这个取舍解释成一条结构化、可测试、纯派生的说明。

## 目标

- 为 `.objectiveDefense` 反制建议新增专用取舍解释模型或 computed property，例如 `objectiveDefenseTradeoff`。
- 解释必须只从现有 preview 字段派生：`destination`、`threatTargetCoordinate`、`targetName`、`routeCost`、`score`、`impactComparisons`。
- 区分两类结论：
  - `destination == threatTargetCoordinate`：进驻据点，强调直接占位、阻止夺点。
  - `destination` 与目标据点相邻：封堵据点，强调相邻卡位、保留路线弹性或无法直达时的防线选择。
- 在反制建议行中显示该取舍解释；非 `.objectiveDefense` 建议不显示。
- 测试证明解释纯派生、不改变反制排序、limit 行为、点选聚焦、执行桥接和真实执行语义。
- 同步 README、flow、flowchart、test、prompt README 和 update_log。

## 非目标

- 不改变 `objectiveDefenseCountermeasure(for:)` 的候选选择、score 公式或排序规则。
- 不新增第二套守点评分，不新增防守 AI、生产、驻防状态或区域控制规则。
- 不新增历史建议列表、动画、自动执行或镜头移动。
- 不让 SwiftUI 计算进驻/封堵、路线、score 或 impact；View 只展示 `GameModels`/`GameState` 给出的字段。
- 不引入第三方库、图片资源或复杂资产管线。

## 当前架构依据

- `EnemyThreatCountermeasurePreview` 已包含 `kind`、`destination`、`threatTargetCoordinate`、`targetName`、`routeCost`、`score` 和 `impactComparisons`。
- `.objectiveDefense` 已由 `GameState.objectiveDefenseCountermeasure(for:)` 生成，候选排序已经稳定。
- `EnemyThreatCountermeasureRow` 已展示反制建议的收益、impact、priority 和 reason，适合加入一条只读取舍说明。
- `GameStateTests.swift` 和 `RulesSmokeTest.swift` 已有 `enemyThreatIntentScenario(axisSpent: true)` 中“后方油库”的 `.objectiveDefense` 场景，可复用，不需要新增 fixture。

## 实现步骤

1. 在 `GameModels.swift` 新增轻量模型，例如 `EnemyThreatObjectiveDefenseTradeoff`：
   - `actionTitle`：`进驻` 或 `封堵`
   - `positionTitle`：`destinationText`
   - `routeTitle`：`路线 N` 或 `无需移动`
   - `impactTitle`：从 objective impact 派生，例如 `被夺风险 -> 进驻`
   - `summary`：适合 UI 展示的一句话
   - `id` 稳定基于以上字段
2. 在 `EnemyThreatCountermeasurePreview` 增加 computed property：
   - `var objectiveDefenseTradeoff: EnemyThreatObjectiveDefenseTradeoff?`
   - `guard kind == .objectiveDefense`，否则返回 nil。
   - 使用 `destination == threatTargetCoordinate` 判断进驻/封堵。
   - 从 `impactComparisons.first { $0.kind == .objective }` 读取 before/after/impact；缺失时给出不执行、不写状态的退化说明。
   - 不新增 stored property，避免更新所有 synthetic initializer。
3. 在 `ContentView.swift` 的 `EnemyThreatCountermeasureRow` 中：
   - 当 `preview.objectiveDefenseTradeoff` 非空时显示一条紧凑 Label。
   - 使用系统图标，例如 `shield.lefthalf.filled`。
   - 文案固定尺寸可换行，不挤压按钮或指标。
   - 增加无障碍说明。
4. 补充 `GameStateTests.swift`：
   - 复用“后方油库”的 `.objectiveDefense`。
   - 断言取舍 action 与 `destination == threatTargetCoordinate` 一致。
   - 断言 summary 包含目标名、路线、优先值或 impact 信息。
   - 断言非 objectiveDefense 建议 `objectiveDefenseTradeoff == nil`。
   - 断言生成取舍解释前后，`enemyThreatCountermeasurePreviews` 排序和 `limit` 结果不变。
   - 断言聚焦该建议后，执行桥接仍为 MOVE、目的地不变、没有生成 replay、单位/据点/指令点不变。
5. 补充 `RulesSmokeTest.swift`：
   - 在已有 objectiveDefense 断言旁追加取舍解释断言。
   - 确认非守点建议不生成取舍解释。
   - 确认聚焦守点建议后执行桥接和地图标记仍保持原语义。
6. 同步文档和日志。

## 验收标准

- `.objectiveDefense` 建议显示进驻或封堵取舍解释。
- 取舍解释字段与 `destination`、`threatTargetCoordinate`、`targetName`、`routeCost`、`score`、objective impact 一致。
- 非 `.objectiveDefense` 建议不显示取舍解释。
- 反制建议排序、limit 结果、聚焦、执行桥接、地图标记和真实执行语义不变。
- 读取取舍解释不改变单位、据点、指令点、战报、message、latest result、AI summary 或胜负。
- SwiftUI View 不推导守点评分或路线，只展示 preview 已有字段。
- 本地轻量检查通过，提交并 push 到 `origin/main`。
- Agent C 下载最新 run artifact，核对 manifest、run attempt、JUnit、规则 smoke 日志、Xcode build 日志和 `.xcresult`。
